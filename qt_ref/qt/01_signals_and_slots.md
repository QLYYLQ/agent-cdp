# Signals & Slots in Qt 6.10.2

> Source: https://doc.qt.io/qt-6/signalsandslots.html

## Introduction

Qt's signals and slots mechanism provides an alternative to traditional callback-based communication between objects. Rather than using function pointers, Qt enables objects to communicate through a type-safe messaging system.

## Core Concepts

### Signals and Slots Overview

"A signal is emitted when a particular event occurs." Signals notify other objects about state changes. "A slot is a function that is called in response to a particular signal." Both are available in classes inheriting from `QObject`.

Key characteristics:
- Type-safe: signal and slot signatures must be compatible
- Loosely coupled: senders don't know about receivers
- Flexible: multiple signals can connect to one slot, and vice versa
- Can connect signals directly to other signals

### Signals

Signals indicate state changes and are automatically generated by the `moc` (Meta-Object Compiler). They:
- Are public access functions
- Should only be emitted from their defining class or subclasses
- Execute connected slots immediately by default
- Are invoked in connection order when multiple slots exist
- Must not be implemented in `.cpp` files

### Slots

"A slot is called when a signal connected to it is emitted." Slots:
- Are normal C++ member functions
- Can be called directly or via signals
- Can be virtual
- Are accessible via signal connections regardless of access level
- Follow standard C++ calling rules when invoked directly

## Implementation Examples

### Basic Counter Class

```cpp
#include <QObject>

class Counter : public QObject
{
    Q_OBJECT

public:
    Counter() { m_value = 0; }
    int value() const { return m_value; }

public slots:
    void setValue(int value);

signals:
    void valueChanged(int newValue);

private:
    int m_value;
};
```

### Slot Implementation

```cpp
void Counter::setValue(int value)
{
    if (value != m_value) {
        m_value = value;
        emit valueChanged(value);
    }
}
```

### Establishing Connections

```cpp
Counter a, b;
QObject::connect(&a, &Counter::valueChanged,
                 &b, &Counter::setValue);

a.setValue(12);  // a.value() == 12, b.value() == 12
b.setValue(48);  // a.value() == 12, b.value() == 48
```

## Advanced Features

### Overloaded Signals and Slots

Use function pointers or `qOverload()` to disambiguate:

```cpp
// Using qOverload
connect(comboBox, qOverload<int>(&QComboBox::currentIndexChanged),
        this, &MyClass::handleIndexChanged);

// Using static_cast
connect(comboBox, static_cast<void(QComboBox::*)(int)>
        (&QComboBox::currentIndexChanged),
        this, &MyClass::handleIndexChanged);

// Using lambda
connect(slider, &QSlider::valueChanged,
        this, [lcd](int value) { lcd->display(value); });
```

### Automatic Connection Management

Qt automatically removes connections when either sender or receiver is destroyed. For lambda connections, provide a context object:

```cpp
connect(button, &QPushButton::clicked, this, [this]{ handleClick(); });
```

### Default Arguments

Signal and slot signatures can include default parameters. When using `SIGNAL()` and `SLOT()` macros, "the signature passed to the SIGNAL() macro must not have fewer arguments than the signature passed to the SLOT() macro."

Valid connections:
```cpp
connect(sender, SIGNAL(destroyed(QObject*)),
        this, SLOT(objectDestroyed(QObject*)));
connect(sender, SIGNAL(destroyed(QObject*)),
        this, SLOT(objectDestroyed()));
connect(sender, SIGNAL(destroyed()),
        this, SLOT(objectDestroyed()));
```

Invalid connection:
```cpp
// Won't work - slot expects QObject* but signal won't send it
connect(sender, SIGNAL(destroyed()),
        this, SLOT(objectDestroyed(QObject*)));
```

### Real-World Example: LcdNumber Widget

```cpp
#ifndef LCDNUMBER_H
#define LCDNUMBER_H

#include <QFrame>

class LcdNumber : public QFrame
{
    Q_OBJECT

public:
    LcdNumber(QWidget *parent = nullptr);

signals:
    void overflow();

public slots:
    void display(int num);
    void display(double num);
    void display(const QString &str);
    void setHexMode();
    void setDecMode();
    void setOctMode();
    void setBinMode();
    void setSmallDecimalPoint(bool point);
};

#endif
```

## Integration with Third-Party Libraries

### CMake Configuration

```cmake
target_compile_definitions(my_app PRIVATE QT_NO_KEYWORDS)
```

### qmake Configuration

```
CONFIG += no_keywords
```

When `no_keywords` is set, replace Qt moc keywords with macros:
- `signals` → `Q_SIGNALS` or `Q_SIGNAL`
- `slots` → `Q_SLOTS` or `Q_SLOT`
- `emit` → `Q_EMIT`

### Library Best Practices

Qt-based libraries should use `Q_SIGNALS` and `Q_SLOTS` in public APIs. Libraries can enforce this by defining `QT_NO_SIGNALS_SLOTS_KEYWORDS` during compilation.

## Performance Considerations

"Emitting a signal that is connected to some slots, is approximately ten times slower than calling the receivers directly, with non-virtual function calls." This overhead reflects the cost of locating connections, safely iterating over receivers, and marshalling parameters generically. For most applications, this overhead is negligible compared to system calls or memory operations.

## Requirements

All classes containing signals or slots must:
1. Include the `Q_OBJECT` macro at the declaration start
2. Derive (directly or indirectly) from `QObject`
